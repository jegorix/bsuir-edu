- если вы находитесь в корне репозитория, используйте пути вида `osisp/lab-4-osisp`
- если вы не в корне репозитория, просто подставьте свой путь к каталогу `osisp/...`
- почти везде достаточно `gcc` и `make`
- для интерактивных и сетевых лаб удобнее держать открытыми 2 терминала

## Общая схема

Для большинства лабораторных шаги одинаковые:

```bash
cd osisp/<имя-лабы>
make
```

Если конкретный `Makefile` поддерживает release-сборку, можно использовать:

```bash
make MODE=release
```

## Lab 1 • `dirwalk`

Рекурсивный обход каталога с фильтрацией по типу объекта и с опциональной сортировкой.

### Быстрый старт

```bash
cd osisp/lab-1-osisp/lab1
make
./dirwalk . -s
```

Полезные варианты запуска:

```bash
./dirwalk . -f -s
./dirwalk . -d -s
./dirwalk . -l -s
./dirwalk -h
```

Для автоматической проверки:

```bash
./test.sh
```

### Как демонстрировать

1. Показать справку:

```bash
./dirwalk -h
```

2. Показать полный обход текущего каталога:

```bash
./dirwalk . -s
```

3. Показать только обычные файлы:

```bash
./dirwalk . -f -s | head -n 20
```

4. Сравнить поведение с `find`:

```bash
find . -type f | sort | head -n 20
./dirwalk . -f -s | head -n 20
```

5. При необходимости подтвердить корректность через скрипт:

```bash
./test.sh
```

### Что проговорить на защите

- программа поддерживает фильтры `-l`, `-d`, `-f`
- `-s` включает сортировку по `LC_COLLATE`
- результат можно сверять с `find`

## Lab 2 • Parent/Child и окружение

Родительский процесс печатает своё окружение, затем по команде порождает дочерние процессы и передаёт им окружение, сформированное по списку переменных из файла `env`.

### Быстрый старт

```bash
cd osisp/lab-2-osisp/lab2
make
./parent
```

Файл `env` задаёт, какие переменные попадут в дочерний процесс.

### Управление

- `+` — запустить дочерний процесс и вывести окружение через `environ`
- `*` — запустить дочерний процесс и вывести окружение через `envp`
- `q` — завершить родительский процесс

### Как демонстрировать

1. Запустить программу:

```bash
./parent
```

2. Показать, что родитель сначала печатает отсортированное окружение.

3. Нажать `+` и показать вывод дочернего процесса.

4. Обратить внимание, что дочерний процесс получает не всё окружение подряд, а только переменные, перечисленные в `env`.

5. Нажать `*` и показать второй способ доступа к тому же окружению, но уже через `envp`.

6. Завершить работу через `q`.

### Что проговорить на защите

- родитель сортирует своё окружение
- дочернее окружение собирается вручную на основе файла `env`
- демонстрируются два способа чтения окружения в дочернем процессе: `environ` и `envp`

## Lab 3 • Управление дочерними процессами

Родительский процесс создаёт и удаляет дочерние процессы, умеет включать и выключать их вывод и печатает список активных процессов.

### Быстрый старт

```bash
cd osisp/lab-3-osisp-main
make
./build/debug/parent
```

### Управление

- `+` — создать дочерний процесс
- `-` — удалить последний дочерний процесс
- `l` — показать список процессов и статус вывода
- `t` — переключить вывод выбранного дочернего процесса
- `k` — завершить все дочерние процессы
- `q` — завершить программу

### Как демонстрировать

1. Собрать и запустить `parent`.

2. Нажать `+` два раза и показать сообщения о создании процессов.

3. Нажать `l` и показать список дочерних процессов со статусами `ON/OFF`.

4. Нажать `t`, ввести номер процесса и показать, что вывод для него переключается.

5. Подождать несколько секунд и показать статистические строки, которые печатают дочерние процессы.

6. Нажать `-` и показать удаление последнего дочернего процесса.

7. Нажать `k`, затем `q`.

### Что проговорить на защите

- управление делается из одного родительского процесса
- дочерние процессы периодически печатают статистику
- вывод конкретного дочернего процесса можно включать и выключать

## Lab 4 • Producer/Consumer на процессах

Реализация задачи producer/consumer через отдельные процессы, кольцевую очередь и семафоры System V.

### Быстрый старт

```bash
cd osisp/lab-4-osisp
make
./build/debug/semaphores
```

Если нужна release-сборка:

```bash
make MODE=release
./build/release/semaphores
```

### Управление

- `1` или `p` — добавить producer
- `2` или `o` — удалить producer
- `3` или `c` — добавить consumer
- `4` или `x` — удалить consumer
- `5` или `s` — показать состояние очереди
- `0` или `q` — завершить программу

### Как демонстрировать

1. Запустить программу.

2. Добавить одного producer.

3. Добавить одного consumer.

4. Подождать несколько секунд и показать строки добавления и извлечения сообщений.

5. Нажать `s` и показать текущее состояние очереди:
   размер, занято, свободно, число producer и consumer.

6. Добавить ещё одного producer или consumer, чтобы показать параллельную работу.

7. Завершить программу через `q`.

### Что проговорить на защите

- producer и consumer работают как отдельные процессы
- очередь защищена семафорами и механизмом взаимного исключения
- видны счётчики добавленных и извлечённых сообщений

## Lab 5 • Producer/Consumer на потоках

В одной программе реализованы два режима:
- `5.1` — POSIX semaphores + mutex
- `5.2` — condition variables + mutex

### Быстрый старт

```bash
cd osisp/lab-5-osisp
make
./build/debug/semaphores
```

Если нужна release-сборка:

```bash
make MODE=release
./build/release/semaphores
```

### Главное меню

- `1` — режим 5.1
- `2` — режим 5.2
- `q` — выход из программы

### Управление внутри режима

- `p` — добавить producer
- `P` — удалить последний producer
- `c` — добавить consumer
- `C` — удалить последний consumer
- `+` — увеличить размер очереди
- `-` — уменьшить размер очереди
- `s` — показать состояние очереди
- `q` — завершить текущий режим и вернуться в главное меню

### Как демонстрировать

1. Запустить программу.

2. Зайти в режим `1`.

3. Добавить producer и consumer.

4. Несколько раз нажать `+` и `-`, чтобы показать динамическое изменение размера очереди.

5. Нажать `s` и показать состояние структуры.

6. Выйти из режима через `q`.

7. Повторить те же действия в режиме `2`.

### Что проговорить на защите

- логика задачи одна и та же, меняется только механизм синхронизации
- в этой лабе уже используются потоки, а не процессы
- можно показать одинаковое поведение в двух режимах и сравнить реализацию

## Lab 6 • Генерация, просмотр и сортировка файла

Набор утилит для работы с индексным файлом:
- `gen` — создать файл
- `view` — просмотреть файл
- `sort_index` — многопоточно отсортировать файл
- `lab6_menu` — интерактивное меню

### Быстрый старт

```bash
cd osisp/lab-6-osisp
make
./build/debug/gen sample.bin 4096
./build/debug/view sample.bin first10
./build/debug/sort_index 16384 16 4 sample.bin
./build/debug/view sample.bin first10
```

Пояснение параметров для `sort_index`:

- `16384` — размер буфера `memsize`
- `16` — число блоков `blocks`
- `4` — число потоков `threads`
- `sample.bin` — входной файл

Ограничения:

- число записей для `gen` должно быть кратно `256`
- `blocks` должно быть степенью двойки
- `blocks >= 4 * threads`
- `memsize` должен быть кратен размеру страницы

### Альтернатива: интерактивное меню

```bash
./build/debug/lab6_menu
```

### Как демонстрировать

1. Сгенерировать файл:

```bash
./build/debug/gen sample.bin 4096
```

2. Показать первые записи до сортировки:

```bash
./build/debug/view sample.bin first10
```

3. Отсортировать файл:

```bash
./build/debug/sort_index 16384 16 4 sample.bin
```

4. Снова показать первые записи после сортировки:

```bash
./build/debug/view sample.bin first10
```

5. При необходимости проверить корректность сортировки:

```bash
./build/debug/view sample.bin | tail -n +4 | awk '{print $1}' | sort -c
```

### Что проговорить на защите

- файл сначала генерируется в несортированном виде
- сортировка выполняется многопоточно
- результат удобно показать через `view` до и после сортировки

## Lab 7 • Конкурентный доступ к файлу записей

Работа с файлом фиксированных записей под управлением блокировок `fcntl()`.

### Быстрый старт

```bash
cd osisp/lab-7-osisp
make
cd build/debug
./gen 10
./records_work
```

### Команды внутри программы

- `lst` — показать все записи
- `get N` — загрузить запись с номером `N`
- `mod` — изменить загруженную запись
- `put` — сохранить запись обратно в файл
- `inf` — показать рабочую, исходную и текущую файловую версии
- `q` — выйти

### Как демонстрировать

Лучший сценарий показывает конфликт между двумя процессами. Для этого нужны два терминала.

Терминал 1:

```bash
cd osisp/lab-7-osisp
make
cd build/debug
./gen 10
./records_work
```

Терминал 2:

```bash
cd osisp/lab-7-osisp/build/debug
./records_work
```

Дальше сценарий:

1. В обоих терминалах выполнить `get 1`.

2. В первом терминале выполнить `mod`, затем `put`.

3. Во втором терминале выполнить `inf`, чтобы показать расхождение между локальной и актуальной версией.

4. Во втором терминале попытаться выполнить `put` и показать, что программа обнаруживает изменение записи другим процессом.

5. Выполнить `lst` для наглядности.

### Что проговорить на защите

- чтение и запись защищены файловыми блокировками
- запись выполняется только после повторной проверки актуальности данных
- конфликт между двумя процессами можно показать вживую

## Lab 8 • TCP сервер и клиент

Простой файловый TCP-сервер с поддержкой команд `INFO`, `LIST`, `CD`, `ECHO`, `QUIT` и пакетного ввода команд из файла.

### Быстрый старт

Терминал 1:

```bash
cd osisp/lab-8-osisp
make
./build/debug/myserver ./testroot 23456
```

Терминал 2:

```bash
cd osisp/lab-8-osisp
./build/debug/myclient 127.0.0.1 23456
```

### Основные команды клиента

- `INFO` — информация о сервере
- `LIST` — список объектов в текущем каталоге
- `CD <dir>` — перейти в подкаталог
- `ECHO <text>` — отправить текст и получить его обратно
- `QUIT` — закрыть сессию
- `@commands.txt` — выполнить команды из файла

### Как демонстрировать

1. Запустить сервер на каталоге `./testroot`.

2. Подключиться клиентом.

3. Ввести команды вручную:

```text
INFO
LIST
CD dir1
LIST
CD ..
ECHO socket lab
QUIT
```

4. При желании показать пакетный режим:

```text
@commands.txt
```

### Что проговорить на защите

- у каждого клиента свой текущий каталог внутри серверного root
- клиент не должен выходить за пределы корневого каталога сервера
- сервер и клиент удобно показывать в двух отдельных терминалах
